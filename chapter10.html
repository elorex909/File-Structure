Binary Searching

What is Binary Search?

A technique for searching an ordered list in which we first check the middle item and - based on that comparison - "discard" half the data. The same procedure is then applied to the remaining half until a match is found or there are no more items left.

Sequential Search
  • Time Complexity: O(n)
  • Checks each record one by one
  • Works on unsorted files
  • Best case: 1 comparison
  • Worst case: n comparisons
  • Average case: n/2 comparisons
Binary Search
  • Time Complexity: O(log n)
  • Divides search space in half each step
  • Requires sorted file by key value
  • Best case: 1 comparison
  • Worst case: log₂(n) comparisons
  • If file size doubles, search time increases by 1

Key Sorting

Key Sorting Concept

When a file is too large to fit into main memory but needs to be sorted:

  • Bring only the keys and their RRNs to main memory
  • Perform internal sorting of keys in memory
  • Rewrite the file in sorted order or create an index

Index Structures

Introduction to Indexing

An index lets us impose order on a file without rearranging the file.

  • Indexes provide multiple access paths to a file
  • Allow keyed access to variable-length record files
  • Simple indexes use arrays in main memory
  • Like a library catalog with multiple search options

How Index Search Works

  1. Binary search in the index (in main memory)
  2. Find key → get reference field (byte offset)
  3. Seek to record position using reference field
  4. Access record directly from disk

Index Maintenance Issues

Record Addition
Append data file + insert in index
Record Deletion
Delete from data file + remove from index
Record Update
Delete + insert if key changes

Persistent Index: Storing index in a file when not in main memory

Safeguards: Status flag in index header + reconstruction procedure

Large Index Management

Indexes Too Large for Memory

When indexes don't fit into main memory:

  • Binary searching occurs on disk (multiple seekg() calls)
  • Index rearrangement requires shifting on disk
  • Performance decreases significantly

Two Main Alternatives:

Tree-Structured Index

B-trees and B+ trees

Hashed Organization

Direct key addressing

Multiple Key Access

Secondary Key Indexing

Primary index allows access by primary key only. Secondary indexes provide access by other fields.

Secondary Key Operations

Record Addition:

Add entry to secondary key index, store in canonical form, keep duplicates sorted by primary key

Record Deletion:

Alternative: Delete from data file and primary index only, leaving secondary indexes unchanged

Record Update:

1. Change secondary key → rearrange secondary index
2. Change primary key → update primary index + secondary references
3. Change other fields → no index change unless record location changes

Key Takeaways

  • Binary search provides O(log n) performance but requires sorted files
  • Key sorting brings only keys to memory for efficient sorting of large files
  • Indexes impose order without rearranging files, providing multiple access paths
  • A simple index consists of sorted key-reference pairs in memory
  • Index maintenance (add/delete/update) must keep index synchronized with data file
  • Large indexes that don't fit in memory require tree structures or hashing
  • Secondary indexes enable access by multiple keys (composer, artist, label, etc.)
  • Secondary key operations have special considerations for efficiency
  • Persistent indexes need safeguards like status flags and reconstruction procedures
  • Indexing is more efficient than file sorting when dealing with pinned records
