Data Compression Methods

Lossless Compression, Entropy, Run-length Encoding, Huffman Coding, and Lempel-Ziv Codes

Chapter Overview

This chapter explores data compression techniques that reduce file size for storage and transmission efficiency. It covers the mathematical foundation of compression (entropy), lossless vs. lossy methods, and three key lossless algorithms: Run-length Encoding, Huffman Coding, and Lempel-Ziv Codes.

Why Compress Data?

Key Benefits

  • Less storage space required
  • Faster transmission over networks
  • Decreased access time for reading/writing
  • Faster sequential processing
  • Optimized resource utilization
  • Reduced I/O bus load when compression/decompression done in I/O processor

Data Compression Methods

Lossless Methods
  • Original = Decompressed (exact same data)
  • Redundant data removed during compression
  • Used for: Legal/medical documents, computer programs
  • Examples:
  • • Run-length Encoding (RLE)
  • • Huffman Coding
  • • Lempel-Ziv Codes (LZ78, LZW)
  • Used in: ZIP, PNG, GIF files
Lossy Methods
  • Original ≠ Decompressed (some data lost)
  • Removes "less important" data
  • Used for: Images, audio, video where perfection isn't critical
  • Examples:
  • • JPEG (images)
  • • MPEG (video)
  • • MP3 (audio)
  • Higher compression ratios than lossless
  • Not covered in detail in this chapter

Entropy: The Math Behind Compression

What is Entropy?

Entropy measures the information content in a message. Higher entropy = more information.

Entropy Formula:

H(x) = -p(x) · log₂ p(x)

Where:
• p(x) = probability of symbol x appearing
• H(x) = information carried by symbol x (in bits)
• log₂ = logarithm base 2 (bits)

Key Insights

  • Rare symbols → High probability → Low entropy → Carry more information
  • Common symbols → Low probability → High entropy → Carry less information
  • The negative sign makes entropy positive (log of probabilities 0-1 is negative)
  • Total entropy = Sum of H(x) for all symbols = Average information per symbol
  • Entropy value represents the theoretical minimum bits per symbol for optimal compression

Example Calculation

Message: A A A B B C (6 symbols total)

Symbol Count Probability p(x) log₂ p(x) Entropy H(x)
A 3 3/6 = 0.5 log₂(0.5) = -1 -0.5 × (-1) = 0.5 bits
B 2 2/6 ≈ 0.333 log₂(0.333) ≈ -1.585 -0.333 × (-1.585) ≈ 0.523 bits
C 1 1/6 ≈ 0.167 log₂(0.167) ≈ -2.585 -0.167 × (-2.585) ≈ 0.432 bits

Total Entropy (H):

H = H(A) + H(B) + H(C) = 0.5 + 0.523 + 0.432 ≈ 1.455 bits/symbol

This is the theoretical limit: Optimal compression needs at least 1.46 bits per symbol on average. Huffman coding tries to achieve this limit.

How RLE Works

RLE Algorithm:

  • Read pixels sequentially
  • When same value repeats >1 time, substitute with 3 bytes:
  • 1. Special code (e.g., 0xFF = run-length indicator)
  • 2. Pixel value being repeated
  • 3. Count of repetitions

RLE Limitations

  • No guaranteed space savings - sometimes compressed file is LARGER than original
  • Worst case: No repeated symbols → Each symbol becomes 3 bytes instead of 1
  • Example: "22 23 24 25 26" becomes "22 23 24 25 26" (no benefit)
  • Best for: Images with large uniform areas (black/white docs, simple graphics)
  • Used in: PCX, BMP (optional), early fax machines

Huffman Coding

Core Principle

Assign shorter codes to frequent symbols, longer codes to rare symbols.

Make a leaf node for each symbol with its probability
Take two nodes with smallest probabilities, connect them
Assign 0/1 to branches, new node probability = sum of children
Repeat step 2 until only one node remains (root)
Read codes from root to leaves (0 for left, 1 for right)

Important Properties:

  • No unique Huffman code - multiple valid trees possible
  • All Huffman codes have same average code length
  • Prefix-free code: No code is prefix of another (enables instant decoding)
  • Tries to achieve the entropy limit (optimal compression)

Lempel-Ziv Codes (LZ78)

Dictionary-Based Compression

Builds a dictionary of strings during compression. Replaces repeated strings with dictionary indices.

Start with empty dictionary
Find shortest substring not in dictionary
Add substring to dictionary with next index
Replace substring with (previous index + new character)
Repeat until entire string processed

Example: LZ78 Encoding

Input: a a a b a b b b a a a b a a a a a a a b a a b b

Step Parsed String Dictionary Entry Encoded As
1 a 1: a 0a
2 aa 2: aa 1a
3 b 3: b 0b
4 ab 4: ab 1b
5 bb 5: bb 3b
6 aaa 6: aaa 2a

Key Insight:

Each piece is concatenation of previous piece + new character.
Encoded as: (previous index) + (new character)

LZ78 in Practice

  • Used in: ZIP, UNIX compress, GIF, PDF
  • Adaptive: Dictionary built during compression
  • Both sides need dictionary (built during compression, rebuilt during decompression)
  • Decompression is inverse process: rebuild dictionary from encoded stream
  • Excellent for text with repeated phrases/patterns

Key Takeaways

  • Entropy measures information content: H(x) = -p(x)·log₂p(x); rare symbols have higher entropy
  • Lossless compression preserves all data (RLE, Huffman, LZ); Lossy sacrifices some data (JPEG, MPEG)
  • Run-length Encoding replaces consecutive repeats with (symbol + count); simple but can expand data
  • Huffman Coding assigns shorter codes to frequent symbols via binary tree; achieves entropy limit
  • Lempel-Ziv (LZ78) builds dictionary of strings; encodes as (index + new char); used in ZIP, GIF
  • Compression benefits: Less storage, faster transmission, reduced access time
  • No single "best" method - choice depends on data type and requirements
  • Real-world tools combine methods (e.g., ZIP uses LZ77 + Huffman)
